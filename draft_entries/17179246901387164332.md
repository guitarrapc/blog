2025年に続き、2026年もGitHub Actionsを経由した攻撃が継続して報告されています。よくも悪くもGitHubとGitHub ActionsはOSSリポジトリのCI/CDのデファクトスタンダードになっているため、攻撃者からも魅力的な攻撃対象になっています。CircleCIなどの外部CIより、GitHub Actionsのほうがいくつかのポイントで攻撃しやすいため、目立っているのもあるでしょう。

2026年に攻撃の起点となっている外部コントリビューターからのPR(以降、外部PR)を信頼できるか問題について、ここしばらく考えているメモです。

[:contents]

## 外部PRを信頼できるか問題

昨今のサプライチェイン問題の多くは外部PRでCI(GitHub Actions)を動かすことが起点になっています。外部PRを信頼できるかが問題の根っこで「フォークPRに対してCIを実行すること自体がリスク」と言い換えてもいいでしょう。[^1]

フォークPRに対してCIを実行することは、PRを通した任意のコードの挿入を許可するリスクととらえられます。npmに代表されるパッケージシステムによるサプライチェインが広がった現状において、任意コードを挿入された汚染パッケージは利用者に一気に波及するため、攻撃者から見ると任意コードを差し込む動機とリターンが高いといえます。

OSSプロジェクトの運営者としては、多数の外部コントリビューターから攻撃を見分ける難しさ、マージした後に気づくのが難しいこと、一度でも見逃すと影響が大きいこと、などが積み重なり頑張らないといけないという状況です。残念ながら一度でも悪意ある仕込みが入った外部PRをマージすると、デフォルトブランチが侵害されます。場合によっては、少しだけ安全性を緩めて、さらなるPRでその間隙を突く手法もあります。そうなった後はデフォルトブランチの安全性が失われ、リリースにまで影響する可能性が高くなります。気づかずにリリース(パッケージ配布)まで到達すれば、攻撃者は本来の狙いに到達するまでサプライチェイン侵害を広げ続けます。

セキュリティ全般に言えますが、一度侵入されたときのリスクが高いため、外部PRをありがたく受け入れるのとリスクのバランスが崩れつつある認識しています。

<details><summary>内部コントリビューターの話を割愛する理由</summary>

あえて外部コントリビュータとしましたが、中には[XZ Utilsのバックドア](https://en.wikipedia.org/wiki/XZ_Utils_backdoor)のように、プロジェクトのコントリビューターとして信頼を積み重ねた上で、悪意あるコードを仕込む攻撃もあります。このケースを持ち出すと、OSSプロジェクト自体の信頼性の問題に話が飛んでしまうので、ここではあえて外部コントリビューターに限定して話を進めます。

この話って、ネットワークにおける境界防御の限界と、ポリシーをベースにしたゼロトラストセキュリティの必要性と似ている面があると感じますが、異なるコンテキストが混じるのでここでは触れないことにします。

</details>

外部PRをどこまで信頼するかは、プロジェクトの性質や運営方針によって異なりますが、少なくとも外部PRに対してCIを実行することはリスクがあることを認識しておく必要があります。2026年5月現時点では、外部PRをそもそも受け付けない、あるいは受け付けるにしてもワークフロー実行の承認を慎重にする選択肢が考えられます。LLMによるコード生成の普及もあり外部PRがなくともLLMでコードを用意できうるシーンが増えてきていることもあり、外部PRを重視する比重に変化があるのは事実でしょう。

### 外部PRを受け付けないという選択

そもそも外部PRを受け付けないという選択肢もあります。GitHubは[2026年2月](https://github.com/orgs/community/discussions/187038)にPull Requestの受付を`Collaborators only`に制限するオプションを追加しました。これを選択すると外部PRは受け付けなくできます。

[f:id:guitarrapc_tech:20260527034219p:plain:alt=Pull request permissions]<!--image-1.png-->

開発はあくまでも内部で行い、コードやリリースやIssueは公開というスタイルには向いています。外部PRを受け付けないことで、CIを経由した攻撃のリスクは低減できます。外部PRを受け付けないことでコードを書き換えられる最も大きな口を塞げるというのは選択肢としては十分に魅力的でしょう。外部PRを受け付けないといっても、Issue(場合によってはDiscordなどの外部コミュニケーションチャンネル)を通じて、外部からのフィードバックやバグ報告を受け付けることはできるので、外部との対話をしないわけではありません。

ただ、IssueやIssue Comment経由で起動するGitHub Actionsがあるかは注意しましょう。`contents: write`や`pull_request: write`などの権限を持っていると攻撃する口になります。たとえば、次のようなワークフローは一見よくありそうですが、危険の塊です。

```yaml
on:
  issues:
    types: [opened, edited]  # <- 攻撃で使いやすい
  issue_comment:
    types: [created, edited] # <- さらに攻撃で使いやすい

jobs:
  example:
    permissions:
      contents: write        # コードの書き換えpushを許容してしまう
      pull-requests: write   # PRの作成や更新を許容してしまう
    runs-on: ubuntu-24.04
    steps:
      - name: Do something when an issue is opened or commented
        run: echo "Issue #${{ github.event.issue.number }} was ${{ github.event.action }}"
      - name: Do something when an issue is opened or commented  # <- Issueタイトル経由でスクリプトインジェクション攻撃できる
        run: echo "Issue #${{ github.event.issue.title }} "
```

Issue/IssueコメントはPR経由でコードを改変されるのと同様に、攻撃者が任意のコードを差し込むリスクがあります。外部PRを受け付ける受け付けないに関わらず、ワークフローでむやみとIssue/Issue Commentを使わないようにしましょう。


### 外部PRのワークフロー実行を慎重に許可する選択

外部PRを受け付けるものの、ワークフロー実行の承認をより慎重にする選択肢もあります。GitHubには、フォークPRからのPRでワークフローを実行するためのいくつかのオプションがあります。これを使うと、外部PRでGitHub Actions実行の前に承認を設けるかを調整できます。Orgnizaion(リポジトリ)の `Settings > Actions` を見る[^2]と次のオプションがあります。

- Require approval for first-time contributors who are new to GitHub: GitHubに新規で、かつこのリポジトリでコミット/PRもマージされたことがないユーザーだけがワークフロー実行の承認を必要とする
- Require approval for first-time contributors: このリポジトリでコミット/PRもマージされたことがないユーザーだけがワークフロー実行の承認を必要とする (デフォルト)
- Require approval for all external contributors: 外部コントリビューター全員がワークフロー実行の承認を必要とする

[f:id:guitarrapc_tech:20260527034227p:plain:alt=Approval for running fork pull request]<!--image.png-->

現実的には`Require approval for all external contributors`を選択するしかないでしょう。なぜなら、外部PRを信頼することは難しいことを前提に置くと、一度でもPRでCI実行を許可したら引き続きその人がコミットしたものをでCI実行するのはあまりにも無防備だからです。以前としてPRの内容を確認せずに承認してしまうリスクは残りますが、一度PRをマージさせた後に悪意あるPRを投げてくるようなユーザーに対して意図せずCIが実行されるリスクは減らせます。

> 後述しますがpull_request_tagetイベントを利用しているワークフローは「初回承認」をバイパスするため、このオプションで外部PRによる承認なしCI実行を防げるわけではないことに注意してください。

## pull_request_targetイベントは制御できないと考える

外部PRでしか用いないのに大きなリスクとなっているのが`pull_request_target`イベントです。これまでに判明しているGitHub Actionsを経由した攻撃の多くは、`pull_request_target`イベントを利用しています。個人的には、今でも`pull_request_target`イベントを利用しているワークフローは`pull_request`イベントに差し替えることをおすすめします。

`pull_request_target`イベントは次のような特徴があるため、攻撃をする側から見ると「リポジトリオーナーが認識できていない経路」で攻撃しやすいです。

- 先に紹介した外部PRに対するのRequire approvalをバイパスして実行される
- `pull_request_target`イベント + `actions/checkout`の組み合わせでフォーク先のコードが無条件でチェックアウトされる
- `contents:write`でコード書き換え、`actions/cache`でキャッシュポイゾニングが可能 (キャッシュ経由でデフォルトブランチまで侵入できる)

これまでにも`pull_request_target`イベントを利用した攻撃が複数回報告されていることから、`pull_request_target`イベントが危ない認識は十分広がっており、`contents: write`や`pull_request:write`のようなwrite書き込みをしないことは共通認識でしょう。

しかし2026年5月11日に発生した[TanStackインシデント](https://tanstack.com/blog/npm-supply-chain-compromise-postmortem)はバンドルサイズを計測するために`pull_request_target`イベントを使っており攻撃を防ぐことができませんでした。経過の詳細は記事を読んでいただきたいですが、トリガーが`pull_request_target`イベント、汚染の起点は`actions/checkout`と`actions/cache`でした。キャッシュは本当に制御が難しいです。

<details><summary>TanStackインシデントにみるpull_request_targetの難しさ</summary>

TanStackが攻撃されたときのワークフローとアクションは過去のコミットから確認できます。経過を軽く見てみましょう。

攻撃者は`pull_request_target`イベントを利用している`bundle-size.yml`ワークフローにて`actions/checkout`でフォーク先のコードを展開、`pnpm install + pnpm nx run @benchmarks/bundle-size:build `を実行、`vite_setup.mjs`を実行して`actions/cache`のキャッシュを汚染しました。キャッシュ汚染後、PRのHEADをmain HEADにforce pushしてファイル変更0にしつつPRをクローズ、ブランチが自動削除されています。あとは、正規の開発フローで汚染済みのキャッシュが展開、リリースに混入するのを待つだけです。実際、リリース時に汚染済みのキャッシュが展開され、Trusted Publising経由でnpmに公開されました。

PRとワークフローを見てみましょう。攻撃者は[PR](https://github.com/TanStack/router/pull/7378)を作ります。

起点となったのは、[TanStack/router/.github/workflows/bundle-size.yml](https://github.com/TanStack/router/blob/9f35318c41fcb38d8cb2bf29519860f7dd3a3386/.github/workflows/bundle-size.yml)ワークフローの`pull_request_target`イベントです。このワークフローは`actions/checkout`でフォーク先のコードをチェックアウトし、さらにOrg内部のアクション`tanstack/config/.github/setup@main`を呼び出しています。

```yaml

  # - `comment-pr` runs trusted base-repo code with limited write access to upsert the PR comment.
  pull_request_target:
  # ... 省略

permissions:
  contents: read

# ... 省略

jobs:
  benchmark-pr:
    name: Benchmark PR
    if: github.event_name == 'pull_request_target'
    runs-on: ubuntu-latest
    outputs:
      current_json_b64: ${{ steps.capture.outputs.current_json_b64 }}
    steps:
      - name: Checkout
        uses: actions/checkout@v6.0.1

          ref: refs/pull/${{ github.event.pull_request.number }}/merge

      - name: Setup Tools

        run: pnpm nx run @benchmarks/bundle-size:build --outputStyle=stream --skipRemoteCache

```

呼び出された[Tanstack/config/.github/setup/action.yml](https://github.com/TanStack/config/blob/d8cef73c4209d91ca578274ce6f83770f64752d3/.github/setup/action.yml)アクション(同OrganizationのAction用リポジトリ)で`actions/cache`を実行したため、キャッシュがフォーク先コードで汚染されました。汚染のために`git force push`を繰り返し行っているのも攻撃にしばしばみられる特徴です。

```yaml
name: Setup Tools
description: Action that sets up Node, pnpm, and caching
runs:
  using: composite
  steps:
    - name: Setup pnpm
      uses: pnpm/action-setup@v4.4.0
    - name: Setup Node
      uses: actions/setup-node@v6.3.0
      with:
        node-version-file: .nvmrc
        package-manager-cache: false
    - name: Get pnpm store directory
      shell: bash
      run: |
        echo "STORE_PATH=$(pnpm store path --silent)" >> $GITHUB_ENV
    - name: Setup pnpm cache
      uses: actions/cache@v5.0.4
      with:
        path: ${{ env.STORE_PATH }}
        key: ${{ runner.os }}-pnpm-store-${{ hashFiles('**/pnpm-lock.yaml') }}
        restore-keys: |
          ${{ runner.os }}-pnpm-store-
    - name: Install dependencies
      shell: bash
      run: pnpm install --frozen-lockfile
```

PRでforce pushを多用してpull_request_targetを利用した多重トリガーしたり、最後に差分をなくすなど、攻撃が発覚しにくいように工夫しています。攻撃者の[フォーク](https://github.com/zblgg/configuration)リポジトリは残っているものの、[アクティビティ](https://github.com/zblgg/configuration/activity)を見てもforce pushコミットの内容は確認できなくなっています。残念ながら、第三者が攻撃の内容を追いかけるのは難しい状況です。

### pull_request_targetをどうしても使いたい

ここまでの内容を踏まえて、なお`pull_request_target`を使うなら`actions/checkout`を使うことは避けましょう。どうしてもフォーク先のコードをチェックアウトするなら、GitHub Actions Permissionsで`write`権限を設定したり`actions/cache`を使うのは絶対にやめましょう。それはそもそも論外です。

```yaml
name: Bundle Size

on:
  pull_request_target:
    branches: [main]

jobs:
  example:
    permissions:
      # contents: write      <- これはダメ
      contents: read         <- どうしてもチェックアウトするならreadにとどめる
    runs-on: ubuntu-24.04
    timeout-minutes: 10
    steps:
      # - uses: actions/checkout@v6.0.1 <- これはフォーク側をチェックアウトするので避ける
      #   with:
      #     persist-credentials: false
      # - name: Cache                   <- これは絶対避ける
      #   uses: actions/cache@v5.0.4
      #   key: ${{ runner.os }}-foo-bar-${{ hashFiles('**/some-lock.yaml') }}
      - name: no checkout
        runs: gh issue list .... # 書き込みとか、チェックアウトしない処理だけにしたい
```

### pull_request_taregetは使わない

私は、`pull_request_target`イベントで事故を起こさないのはあまりに難しすぎると考えています。たとえLLMでワークフローの内容を検証したとしても、継続的に事故が起こらないように使い続けられる自信がありません。リモートアクションを使っていたらなおさらです。

`pull_request_target`イベントは、PRのRequire approvalをバイパスして実行されるためワークフローの書き方次第で事故が起きます。権限以外にキャッシュを使っていても問題になります。キャッシュのコンテンツ内容を監査しているOSSは果たしていくつあるかも疑問です。permissionsで`read`を徹底するのも難しいことを考えると、`pull_request_target`は使わないほうがいいです。[^3]

```yaml
on:
  # そもそも pull_request_targetイベントは使わないほうがいい。代わりにpull_requestイベントを使う。
  pull_request:
    branches: [main]
```

TanStackはインシデントを受けてどう対応したかというと、`pull_request_target`から`pull_request`へ変更、かつSetup toolsの`actions/cache`をやめています。この対応は個人的に納得できるものです。

<details><summary>TanStackの対応</summary>

変更後のBundle Size処理を見ると、`actions/cache`の代わりに`$GITHUB_OUTPUT`経由でバンドルサイズを渡してPRへレポートしている。

```yaml
name: Bundle Size

on:
  # <- もうpull_request_targetは使っていない
  pull_request:
    paths:
      - 'packages/**'
      - 'benchmarks/**'
  # ここでは省略しているがpushトリガーもある

# 省略

jobs:
  benchmark-pr:
    name: Benchmark PR
    if: github.event_name == 'pull_request' # <- イベントを縛っている
    runs-on: ubuntu-latest
    outputs:
      current_json_b64: ${{ steps.capture.outputs.current_json_b64 }} # comment-prに結果を渡す
    steps:
      - name: Checkout
        uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v6.0.2
        with:
          ref: refs/pull/${{ github.event.pull_request.number }}/merge
          fetch-depth: 0
          persist-credentials: false

      - name: Setup Tools              # <- もうactions/cacheをしていない
        uses: TanStack/config/.github/setup@e4b48f16568324f76f467aa4c2aac2f05db632c3 # main

      - name: Measure Bundle Size
        run: pnpm nx run @benchmarks/bundle-size:build --outputStyle=stream --skipRemoteCache

      - name: Capture Benchmark Outputs # <- バンドルサイズをエンコードしている
        id: capture
        run: |
          {
            echo "current_json_b64=$(base64 -w 0 < benchmarks/bundle-size/results/current.json)"
          } >> "$GITHUB_OUTPUT"

  comment-pr:
    name: Upsert PR Comment
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    needs: benchmark-pr    # <- benchmark-prの結果を受け取る
    permissions:
      contents: read
      pull-requests: write
    steps:
      - name: Checkout
        uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v6.0.2
        with:
          fetch-depth: 0
          persist-credentials: false

      - name: Restore Benchmark Outputs
        env:
          CURRENT_JSON_B64: ${{ needs.benchmark-pr.outputs.current_json_b64 }} # benchmark-prからバンドルサイズの結果を受け取る
        run: |
          mkdir -p benchmarks/bundle-size/results
          node -e "const fs=require('node:fs'); fs.writeFileSync('benchmarks/bundle-size/results/current.json', Buffer.from(process.env.CURRENT_JSON_B64 || '', 'base64'))"

      - name: Read Historical Data (if available)                # <- ベースラインのバンドルサイズを取得
        run: |
          mkdir -p benchmarks/bundle-size/results
          if git fetch --depth=1 origin gh-pages; then
            if git show origin/gh-pages:benchmarks/bundle-size/data.js > benchmarks/bundle-size/results/history-data.js 2>/dev/null; then
              echo "Loaded bundle-size history from gh-pages."
            else
              rm -f benchmarks/bundle-size/results/history-data.js
              echo "No bundle-size history found on gh-pages yet."
            fi
          fi

      - name: Build PR Report
        env:
          BASE_SHA: ${{ github.event.pull_request.base.sha }}
          REPOSITORY_NAME: ${{ github.event.repository.name }}
          REPOSITORY_OWNER: ${{ github.repository_owner }}
        run: |
          node scripts/benchmarks/bundle-size/pr-report.mjs \
            --current benchmarks/bundle-size/results/current.json \
            --history benchmarks/bundle-size/results/history-data.js \
            --output benchmarks/bundle-size/results/pr-comment.md \
            --base-sha "$BASE_SHA" \
            --dashboard-url "https://${REPOSITORY_OWNER}.github.io/${REPOSITORY_NAME}/benchmarks/bundle-size/"

      - name: Upsert Sticky PR Comment                              # <- PRにコメント
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          node scripts/benchmarks/common/upsert-pr-comment.mjs \
            --pr "${{ github.event.pull_request.number }}" \
            --body-file benchmarks/bundle-size/results/pr-comment.md
```

</details>

### workflow_runイベントという代替手段の難しさ

GitHubは、ベター`pull_reqest_target`として`pull_request`と`workflow_run`イベント組み合わせを[紹介](https://docs.github.com/en/actions/reference/workflows-and-actions/events-that-trigger-workflows#workflow_run)しています。

<details><summary>pull_requestとworkflow_runイベントを組み合わせる例</summary>

`pull_request`イベントでビルドして、`workflow_run`イベントで`Secrets.FOO`などシークレットが必要なリリースするようにすれば、PRのCIで攻撃されるリスクを減らしつつ、マージ後のリリースで攻撃されるリスクも減らせるという建付けです。

```yaml
name: Bundle Size

on:
  pull_request:
    branches: [main]

jobs:
  build:
    permissions:
      contents: read      # <- 権限をミニマムにする
    runs-on: ubuntu-24.04
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v6.0.1    # フォーク側をチェックアウト
      - run: pnpm build
      - uses: actions/upload-artifact@v4 # 別のワークフローはアーティファクトだけ使う
        with:
          name: build-result
          path: ./dist
```

別のワークフローで`workflow_run`イベントをトリガーして、ビルド結果のアーティファクトをダウンロードすれば、直接フォーク先のコードをチェックアウトすることなく、ビルド結果だけを利用できるという考えです。

```yaml
on:
  workflow_run:
    workflows: ["Bundle Size"] # 上のワークフロー名
    types: [completed]

jobs:
  download:
    runs-on: ubuntu-latest
    steps:
      - name: 'Download artifact'
        uses: actions/github-script@v8
        with:
          script: |
            let allArtifacts = await github.rest.actions.listWorkflowRunArtifacts({
               owner: context.repo.owner,
               repo: context.repo.repo,
               run_id: context.payload.workflow_run.id,
            });
            let matchArtifact = allArtifacts.data.artifacts.filter((artifact) => {
              return artifact.name == "pr_number"
            })[0];
            let download = await github.rest.actions.downloadArtifact({
               owner: context.repo.owner,
               repo: context.repo.repo,
               artifact_id: matchArtifact.id,
               archive_format: 'zip',
            });
            const fs = require('fs');
            const path = require('path');
            const temp = '${{ runner.temp }}/artifacts';
            if (!fs.existsSync(temp)){
              fs.mkdirSync(temp);
            }
            fs.writeFileSync(path.join(temp, 'pr_number.zip'), Buffer.from(download.data));

      - name: 'Unzip artifact'
        run: unzip "${{ runner.temp }}/artifacts/pr_number.zip" -d "${{ runner.temp }}/artifacts"

      - run: 何かしらの処理     # <- ここでdist/の中身を利用する>
```

`dist/`にビルド結果以外が仕込まれたりビルド結果に仕込まれると、`dist/`が伝搬して攻撃されるリスクが残ります。この線引きとリスクのバランスをどうするかはプロジェクトの性質や運営方針によるでしょう。

</details>

ただ、`workflow_run`イベントも`pull_request_target`同様に権限が強すぎて同様に危うさがあり、またworkflowがメインブランチ以外もトリガーできるので制御が難しいと感じます。利用はかなり慎重にすべきですし、可能なら使わないほうがいいでしょう。

```yaml
on:
  workflow_run:
    workflows: ["Bundle Size"] # <- ワークフロー名を名前で指定できるため、意図しないワークフローもトリガーできてしまう。ワークフローIDのほうが安全そうなのに...
    types: [completed]         # <- ワークフローが完了したときに呼ばれる。フォークPRを起点としたワークフローか、などの制御がない
    # 必要ならブランチも指定できるので、メインブランチだけという保証もない
```

## PRレビューとエンコード文字や不可視文字

「見落としのないPRレビューは難しい」、これはレビュー経験がある多くの人が同意するところじゃないでしょうか。少なくとも私はbaseNN(base64など)エンコードされた文字列、不可視文字をファイルに紛れ込ませたPRを見落とすリスクを否定できません。とても怖いと考えています。

不可視文字を使ったコード侵入といえば[GlassWormサプライチェイン攻撃](https://qiita.com/nogataka/items/9952280fe6fddfd14ddf)ですが、これを見落とさず防ぐのは割と難しく感じます。現実的には外部PRのCIで検知するしかないでしょう。一般的なコーディングにおいて不可視文字やエンコーディング文字列を使うシーンが限定されるはずなので、それ自体をはじくのも有効になりえます。

対応例:

- 行差分に対して異様な長さの文字列をはじく
- 不可視文字に相当するUnicodeレンジチェック
- base64などのエンコードされた文字列、不可視文字列の検出にLLMを併用して解釈ヒント
- Copilot Code ReviewのようなLLMによるコードレビューをハーネスとして用意する
- CodeQLを有効にする

誤検知もあるでしょうが、外部PRの内容を機械的にチェックする仕組みで人間の見落としを補助します。CIでビルドやリグレッションテスト、パフォーマンス悪化を確認するという以外に、妙なコードが紛れていることを前提にチェックしていくというのは5年前にはあまり積極的に見かけたやり方ではなかったように感じます。CIでビルドしかチェックしない牧歌的なやり方の終焉を予感させます。

とはいえ、できることならGitHub側で検知してはじけるようにしてほしい気もします。CodeQLなどGitHub提供のセキュリティ機能でサポートされると負担が軽減されます[^4]。正直なところ、外部コードをチェックアウトしたくない気持ちがあるため、ユーザー側で外部PRに変な文字列がないか検知するのはあまりやりたくないのが本音です。

## 侵入されたときの緩和策を設定しておく

侵入されないのが一番ですが、コードやアカウントへの侵入に備えておくのも重要です。アカウント侵入されたときのリスクを減らすための対策例は次のようなものが考えられます。

- デフォルトブランチをforce push禁止にして保護する
- TOTPからFIDO2/WebAuthnなどのハードウェアキーへの移行を検討する
- GPG/SSH鍵によるコミット署名を導入して改ざんされたコミットを検知する
- VS Code拡張は最低限のもの以外外し、見おぼえないものがないか定期チェック

汚染されたパッケージがリリースされたときに備えて、次のような対策も考えられます。

- SBOMを有効、公開して、パッケージに何が含まれるかを明示する
- Assestationを有効にして、リリースされたパッケージがどのようなビルドプロセスを経て作られたかを明示する
- GitHubのImmutable Releaseを導入して、過去のリリース内容を改ざんできないようにする

### GitHubのImmutable Releaseは有効にしよう

GitHubのImmutable Release、使っていますか?これ割と強力なので、有効にするのをおすすめします。たとえアカウント侵入に成功してリリースを消すことはできても、同名リリースで汚染パッケージを公開できないので過去のパッケージに遡及的に影響を与えることができません。過去リリースを汚染パッケージでリリースし直すのは、攻撃でよくみられる手口ですが、Immutable Releaseを入れておけばそれもできなくなります。

- リリースがあるとき、リリースに紐づくタグは削除できない
- リリースを消すとタグは削除できる
- 一度リリースを消すと、同名リリースは作れない
- 一度タグを削除すると、同名タグは作れない

Immutable Releaseを有効にしてから出したリリースは、リポジトリ設定でImmutable Releaseをを無効にしてもリリースを上書きできません。動作確認してて嬉しかったことです。素晴らしい。

### GitHub Environmentを使ったpublishのレビュー承認

アカウントを乗っ取られると、リリースフローをトリガーされて誰も見てない間にリリースされる可能性があります。`npm stage`がなくても、GitHub Environmentを使ってリリースフローにレビュー承認を入れるのは有効な対策になりえます。

EnvironmentのReview RequiredをPrevent self-review付きで有効にすることで、自分以外の承認をもらうまでリリースが止まります。承認する人は、意図したリリースか確認してからリリースを承認するという流れです。

[f:id:guitarrapc_tech:20260527185511p:plain:alt=review required]<!--image-2.png-->

[f:id:guitarrapc_tech:20260527185519p:plain:alt=release stop by review required]

難点として、GitHub Actionsで完結しているので、1人開発だとGitHubアカウントが乗っ取られたときは突破されます。後述するCIとリリースを分離できるほうがより安全です。

## GitHubやパッケージマネージャーの課題

外部PRや攻撃に対して、プラットフォーム側での対策が進むと嬉しいものがあります。

### ビルドとリリースの分離

ビルド(GitHub Actions)とリリース(GitHub Release)やパブリッシュ(id-token)が一緒になっているため、トークンの範囲が広がりやすいのも問題の一員です。必要なジョブに必要な権限しかないことを強制できたほうが安全です。

この観点で、`npm stage`は現時点では理想的な[対応](https://github.blog/changelog/2026-05-22-staged-publishing-and-new-install-time-controls-for-npm/)に見えます。[ドキュメント](https://docs.npmjs.com/staged-publishing)を見ると、CIからはステージまでしかリリースできず、ステージから本番リリースはTOTP(2要素認証)を入力しないと発行できない仕組みです。まさにほしいビルドとリリースの分離です。GitHub Actions(CI)はステージまでしかアクセスできないので、GitHubアカウントが乗っ取られてもnpmに展開されません。悪意のあるパッケージを展開するには、GitHubでステージに展開 + npmのアカウント乗っ取りの両方が必要です。

- CIが関与できる範囲: GitHub Actions (CI) -> npm stage (Trusted Publishing)
- npm(リリース)で関与する範囲: npm publish (with 2FA)

現時点でステージングパブリッシュができるのはnpmぐらいで、他のパッケージマネージャーやサイトはそもそもステージという概念を持っていないと理解しています。すくなくともcrate.ioやNuGetは持っていません。ビルドとリリースを分離するには、パッケージマネージャー側にステージ + ステージから本番リリース + 本番リリース時に強めの本人認証、という仕組みが必要でパッケージマネージャー側の対応が必要と理解しています。

### フォークコミットを拾える仕組みの問題

GitHubはフォーク元からフォーク先のコミットを拾えるのですが、かなり厄介な問題です。

TanStack/routerで例を見てみましょう。フォーク先であるsprusr/routerのコミット https://github.com/sprusr/router/commit/60d63821f2a3cc2b9f808da9467ee9618c83e198 があります。
フォーク元のTanStack/routerでこのコミットはまだマージされていないので、本来 https://github.com/TanStack/router/commit/60d63821f2a3cc2b9f808da9467ee9618c83e198 はアクセスできないでほしいのですが、実際はフォーク先のコミットにTanStack/routerリポジトリのコミットURLからアクセスできます。

これが問題になるのが、GitHub Actionsのリモートアクションを[SHAピンニングするプラクティス](https://docs.github.com/en/actions/reference/security/secure-use#using-third-party-actions)です。GitHub Actionsのリモートアクションは、`actions/checkout@v6`のような書き方ではなく`actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v6.0.2` のようにSHAでピンニングして参照することが推奨されています。バージョンタグで指定しているとバージョン側に不意の変更が入った時に意図せず参照するため、SHAでピンニングして影響を緩和するのが目的です。[^5]

しかしリモートアクションをSHAでピンニングしても、フォーク先のコミットはアクセスできます。攻撃者から見ると、攻撃したいリポジトリで利用しているアクションをフォークして、コードをプッシュ。攻撃したいリポジトリのアクションSHAピンを書き換えるという流れで任意のアクションを呼ばせる余地があります。うかつに承認すれば、任意のコードが差し込めそうですね。

```yaml
jobs:
  example:
    runs-on: ubuntu-latest
    steps:
      - uses: example/actions@0123456789abcdefg # imposter commitの例 (example/actionsではなく example111/actionsというフォークコミットを指しているとする)
```

ユーザーができる対策は2つあります。

- Linterによる検知
- GitHub Actionsファイルの変更を許容しない

そもそも外部PRでGitHub Actionsファイルを変更できるのが問題なので、外部PRでGitHub Actionsファイルの変更を許容しないのは妥当でしょう。触らせないは大事。怪しいコミットを検出Linterとして、私が作っている[guitarrapc/seiton](https://github.com/guitarrapc/seiton)や[zizmor](https://github.com/zizmorcore/zizmor)があります。

- seiton: [Imposter Commit](https://github.com/guitarrapc/seiton/blob/main/docs/rules.md#impostor-commit)としてアクションのSHAが、アクションオーナーのコミットであるかを検査するルールを提供
- zizmor: [Imposter Commit](https://docs.zizmor.sh/audits/#impostor-commit)としてアクションのSHAが、アクションオーナーのコミットであるかを検査するルールを提供

[検知に限界](https://zenn.dev/shunsuke_suzuki/scraps/898fc71e98926b)はあるものの、通常リリース時は自リポジトリオーナーのコミットがあることが多いはずなので、ほとんどのケースでは有効に検知するはずです。

## まとめ

- 突き詰めると、GitHub ActionsでリポジトリのCIを実行すること自体がリスクになっている。CIを実行するための安全な方法がないといえる
  - CI専用のリポジトリを用意して、writeを一切除去。public repoは無権限でcloneできることを利用して、そのリポジトリでクローンしてビルドだけするぐらいの仕組みのほうが安全といえる。なぜ安全課というと、仮にそこのリポジトリに侵入、トークンが漏れてもそのリポジトリのreadしかできず、そもそも外部リポジトリへのアクセスができない。当然publishもできないので影響をかなり封じ込められ宇
  - 問題点は、どうやって外部リポジトリのCIをトリガーするか。Webhook? でもそれってJenkinsと同じようなことをやっているような。JenkinsがCI第一世代、GitHub Actionが第二世代なら、やろうとしているのは1.5世代のCIにみえる
  - トリガーが追跡しやすいかどうかは、問題時にどこから侵入されたかを特定するのに重要。GitHub Actionのトリガーは、PRやpushなど、GitHub上で完結しているものが多いが、webhookでトリガーできる、あるいはRepository Dispatchでトリガーできるのは微妙では
- OIDCトークンレスパブリッシュは必ずしも安全を担保するものではないという認識が必要。比較的安全なパターンは、OIDCでトークンを発行、トークン時限以外にjob終了時に自動expireするもの。そうではなく、トークン発行が隠蔽されているものは、job終了時のトークンExpireがない可能性があるので注意が必要。ただし、トークン発行からジョブ終了まで時間がかかるものは、その間に詐取したトークンを利用される余地があるので、トークン発行を利用直前マで遅延させたほうがいい
- GitHub Actionsではなく、インターナルなCIを使うのか
