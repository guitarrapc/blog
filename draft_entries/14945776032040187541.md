GitHub Actionsのセキュリティに気を使って、都度対応するのに疲れてしまいました。セキュリティに沿ったプラクティスを守るため、検出と修正を自動化するツール「seiton」を作りました。

https://github.com/guitarrapc/seiton

LLM向けにSkillも提供しており、seitonを使ってGitHub Actionsのセキュリティを自動化できます。今回はseitonの紹介と、GitHub Actionsのセキュリティに対するどうあってほしいかについて書きます。

[:contents]

## GitHub Actionsの書き方チェックに複数ツールを使わないといけないのは大変

私は100以上のリポジトリを見ており、GitHub Actionsも相応の数があります。これまではactionlint/ghalint/zizmor/pinactを使っていました。GitHub Actionsのワークフローをチェックするのに複数のツールを併用するのが当たり前になっています。しかし複数のツールを併用するのは相当に大変です。どのツールが何を検出して、修正はどうやって、コンフィグはどうやるかなど管理が複雑になります。

例えば[guitarrapc/githubactions-lab](https://github.com/guitarrapc/githubactions-lab)はactionlint/ghalint/zizmorを使っていた時、ツールそれぞれに設定ファイルを用意していました。githubactions-labはGitHub Actionsの悪い例も載せる意図があるため多数検出しやすいのですが、GitHub Actionsを管理するリポジトリであるあるな検出も多くいいベンチマークになります。

## GitHub Actionsのセキュリティに対するどうあってほしいか

複数の静的解析ツールを用いる現状はツールごとに目的や性質が違うため当然起こります。しかし、GitHubに対する攻撃の入り口としてGitHub Actionsが狙われている現状を踏まえると、GitHub Actionsのセキュリティに対する検出と修正はもっとシンプルであってほしいです。また、静的解析ツールで検出されるほとんどは単純な置き換えで済むことが多いのに、それを手でやったり、LLMにやらせるのも精神的につらい作業です。

私は、GitHub Actionsのセキュリティに対する検出と修正は自動化されていて、手修正が必要でも簡単にできる必要があると考えています。つまり欲しいのは、ワークフローとしての妥当性、セキュリティを同時にチェックし、単純な置き換えで修正できるものは自動で修正し、そうでないものもLLMに適切にスキル指示して簡単に修正できるツールです。seitonはこの考え方に基づいて作られています。
