GitHub Actionsのセキュリティに気を使って、都度対応するのに疲れてしまいました。そこで、セキュリティプラクティスに沿ってワークフロー不備検出と修正を自動化するツール「seiton」を作りました。

- リポジトリ: https://github.com/guitarrapc/seiton
- プレイグラウンド: https://guitarrapc.github.io/seiton

LLM向けにSkillも提供しており、seitonを使うことでGitHub Actionsワークフローのセキュリティ対応を自動化できます。
今回はseitonの紹介と今後どのようにアップデートしていくかを書きます。

## GitHub Actionsのセキュリティに対する課題

GitHub Actionsが登場して数年たち、おおむねマネージドCI/CDとしての地位を確立しました。
利用が増えるとともに増えたのが、GitHub Actionsワークフローの設定不備を突いたセキュリティインシデントです。

多くはワークフローを適切に書けば防げるものですが、肝心の**適切に書く**が難しいのはGitHub Actionsが抱える課題です。

### GitHub Actionsワークフローを適切に書く難しさ

GitHub Actionsを適切に書くのは難しいです。そもそもYAMLなのでシンタックスをミスしやすく、インデントを間違えたり、YAML構造としてキーや値に何が入るべきか迷います。LLMに書かせれば「それっぽくちゃんと動くYAML」を生成してくれますが、今度は前情報がないので、「ジョブのパーミッション指定やタイムアウト考慮がない」「ランナーがlatest」「アクションはSHAピンされていない」「pull_request_targetを使う」など過去にGitHub Actions起因の事故が起きたようなワークフローを生成することがほとんどです。

**セキュリティに沿ったGitHub Actionsを書くのが難しい**のは、GitHub Actionsの大きな課題といえます。

### リンターの守備範囲がバラバラの難しさ

GitHub Actionsのワークフローが適切にかけているかチェックするのに利用されているのがリンターです。
GitHub Actionsリンターは多数あるのですが、それぞれの守備範囲が異なるため複数のツールを併用するのが当たり前になっています。

しかし複数のツールを併用するのは相当に大変です。どのツールが何を検出して、修正はどうやって、コンフィグはどうやるかなど管理が複雑になります。
1つのリポジトリでも面倒なのに、リポジトリが10、100、200と増えるほど、複数ツールを使うことに伴う「ルール事情」を勘案することに疲弊します。

私の管理する[guitarrapc/githubactions-lab](https://github.com/guitarrapc/githubactions-lab)リポジトリは、seitonを作るまでは[actionlint](https://github.com/rhysd/actionlint)/[ghalint](https://github.com/suzuki-shunsuke/ghalint)/[zizmor](https://github.com/zizmorcore/zizmor)/[pinact](https://github.com/suzuki-shunsuke/pinact)を併用していました。

当時、リンターそれぞれに設定ファイルを用意していたため、計3つの設定ファイルを管理する必要がありました。
しかも、リンターごとに検出されるルールが違うため、同じワークフローでもactionlintでは検出されるのにghalintでは検出されないなど、ルール事情を把握するのが大変でした。

本リポジトリは特にワークフローが多いのですが、ワークフローが少なくても多数のリポジトリを管理しているなら、リンター事情を考慮して設定するのは大変です。

## GitHub Actionsのセキュリティに対するどうあってほしいか

現状はツールごとに目的や性質が違うため、複数の静的解析ツールを用いるのは当然です。しかし、GitHubに対する攻撃の入り口としてGitHub Actionsが狙われている現状を踏まえると、リンターによるセキュリティ検出と修正はもっとシンプルであってほしいです。また、静的解析ツールで検出されるほとんどは単純な置き換えで済むことが多いものの、修正は手作業やLLMになっています。なぜその指摘が出たのか、修正する背景やどうあるべきなのかをLLMに伝えるのも、あまりリンターでサポートされていません。

私は、セキュリティに沿ったGitHub Actionsワークフローへ修正するのを自動化したいと考えています。具体的には以下が欲しいです

- GitHub Actionsのワークフローとしての妥当性をチェックする
- ワークフローのセキュリティ考慮が十分か検出する
- 検出された簡単な修正は自動化される
- 自動修正できなくてもLLMが自動修正するか判断するヒントが十分に与えられる
- 無視すべき検出を無視するための設定が簡単にできる

つまり欲しいのは、ワークフローとしての妥当性、セキュリティを同時にチェックし、単純な置き換えで修正できるものは自動で修正し、そうでないものもLLMに適切にスキル指示して簡単に修正できるツールです。

seitonはこの考え方に基づいて作っています。

## seitonの特徴

seitonは、linter兼fixerとして、GitHub Actionsワークフローのセキュリティ対応を自動化するツールです。基本的なワークフローはLLMにスキル指示して修正することを想定しているため、LLMにスキル指示するための情報を十分に出力することを重視しています。

スキルは次のコマンドでインストールできます。

```bash
seiton install --skills
```

以降はLLMにスキル指示して修正するといいでしょう。

```txt
```

もちろんCLIとして、人間が検出、修正できます。

```bash
# 検出
seiton
# コンフィグの生成 & 長征
seiton init
# 修正
seiton --fix

# コンフィグなしでpinning含むネットワーク修正を行う
seiton --fix --enable-pin-network --enable-image-network
```

### GitHub Actionsの定義更新への追随

seitonは、GitHub Actionsの定義、公式アクション(`actions/*`など)のセキュリティ対応が入った時に、素早く追随したいと考えています。
seitonは、GitHub Actionsのドキュメントから定義を生成し、公式アクションの変更を追跡して型定義を生成する仕組みを備えています。

先日1.1.0をリリースしましたが、ここにはactions/checkout@v7で入った、`untrusted-pr-checkout: bool`パラメーターの検出ルールが追加されています。

## 既存リンターへのコントリビューションを選ばなかった理由

複数リンターを使っていたので、seitonを作る前に他ツールへのコントリビューションで自動修正やLLMスキル指示の機能を追加できないか検討しました。
しかし、actionlint/ghalint/zizmor/pinactはそれぞれの目的が異なるため、セキュリティ対応の自動修正やLLMスキル指示の機能を追加するのは難しいと判断しました。

### 方向性の違い

最も大きいのは、ツールそれぞれの目指す方向性の違いです。

actionlintはゼロコンフィグでいい感じに動くことを目指しているため、積極的なセキュリティ的な検出は目指していないと明言されています。
この方針は個人的にはとても好きですが、残念ながらGitHub Actionsの周辺事情はよりセキュリティにフォーカスする必要があるため、actionlintっぽい使い勝手だけどセキュリティにフォーカスしたツールが欲しいと考えました。

ghalintはミニマムなリンターと明言されており、ルールの追加はそもそもツールの目指すところではありません。
そこにルールを足したいというのは、ツールの意図とずれており同意も得られないでしょう。

zizmorは最もseitonと近く、いくつかのルールは自動修正にも対応します。ただペルソナである程度制御できるとはいえ、デフォルト検出が激しく、抑制のためのコンフィグも煩雑になりやすいです。
このツールと一緒にセキュリティ対応を続けていくのは相応に大変で、疲弊した原因でもあります。

pinactは、GitHub Actionsのパーミッションを検出することに特化しており、セキュリティ対応の自動修正やLLMスキル指示の機能を追加するのは難しいです。
そもそもSHAピンニングはセキュリティ対応上、暫定で必要になっているプラクティスです。ピンニングだけのためにツールを増やすのは避けたいです。

### Agentic Workflowへの対応

もう1つ既存ツールが難しい理由はAgentic Workflowへの対応です。最近GAになったAgentic Workflowは、マークダウンからgh cliでワークフローを生成するため、ワークフローのYAMLをいじることなくそのままにする必要があります。
しかし残念ながらAgentic Workflowの生成YAMLは、タイムアウト欠如やshaピンニングなしなど、各種リンターで引っかかりやすい記述内容になっています。

既存のリンターはAgentic Workflowの存在を想定せずひっかけるため、そのまま使うと大量の警告が検出されます。

actionlintは、Agentic Workflowで検出された各文言を逐次除外するようにコンフィグ調整が必要です。先の例を見ても、数が増えた時や、再コンパイルで文言などが変わったら再調整が必要になります。

ghalintは、一見するとルールが少なくコンフィグも柔軟に見えます。しかし、実際のところコンフィグは意図的に限定されており、一部ルールは無効化の口がありません。
このため、Agentic Workflowの生成YAMLがあると、ghalintは警告を出し続けてCIで使っていくのが難しくなります。

zizmorは、コンフィグにignoreを指定することでAgentic Workflowの生成YAMLを触ることなく除外できます。
ただ、コンフィグでの除外は、ルール毎にファイルを指定する必要があるため、Agentic Workflowの生成YAMLが1つでもコンフィグが煩雑です。複数になったら割と地獄です。
